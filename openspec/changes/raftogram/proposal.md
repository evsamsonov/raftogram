## Why

**Название продукта:** Raftogram.

Распределённый мессенджер требует согласованного состояния чатов при сбоях узлов и сетевых разделениях. Алгоритм Raft даёт понятную модель лидерства и линейной репликации лога, что упрощает корректность по сравнению с ad-hoc синхронизацией. Сейчас в репозитории нет прикладного кода — возможность заложить архитектуру «консенсус + state machine» с самого начала.

## What Changes

- Вводится серверный кластер с репликацией состояния через Raft (выборы лидера, append entries, безопасные чтения с лидера или с кворумом — по дизайну).
- Состояние мессенджера (каналы/диалоги, сообщения, порядок) моделируется как детерминированная state machine поверх согласованного лога команд.
- Клиентский протокол для регистрации сессии, отправки сообщений и подписки на обновления (транспорт уточняется в дизайне).
- Наблюдаемость и эксплуатация: метрики узла, health, базовые ограничения ресурсов (без превращения в полноценный SRE-пакет на первом этапе).

## Capabilities

### New Capabilities

- `raft-cluster`: формирование кластера, членство, выборы лидера, репликация лога, снапшоты и восстановление последователей.
- `messenger-state-machine`: команды домена (например: создать канал, отправить сообщение, доставить историю), детерминированное применение к состоянию, инварианты порядка и идемпотентности там, где нужно.
- `client-messaging-api`: контракт клиента к кластеру (аутентификация/идентификация минимум для прототипа), отправка и получение сообщений, модель ошибок и повторов.

## Impact

- Новый код сервера (ориентир: Go по правилам репозитория) и, при необходимости, отдельный клиент или SDK.
- Зависимости: сеть между узлами, персистентное хранилище лога и снапшотов на каждом узле.
- API и схемы данных для клиентов; конфигурация кластера (стартовые пиры, идентификаторы узлов).
