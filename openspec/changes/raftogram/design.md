## Context

Репозиторий сейчас содержит в основном OpenSpec-артефакты; прикладного сервера нет. Цель — **Raftogram**: распределённый мессенджер, в котором **единый упорядоченный лог команд** реплицируется через Raft, а **прикладное состояние** получается детерминированным применением этих команд (state machine). Предполагаемый стек сервера — **Go** с принятой в проекте слоистой архитектурой, `context`, структурированным логированием (zap).

## Goals / Non-Goals

**Goals:**

- Кластер из нечётного числа узлов с устойчивостью к падению меньшинства узлов и корректным лидерством Raft.
- Репликация и коммит команд домена мессенджера; чтение согласованного состояния согласно выбранной модели (лидер-чтения для прототипа допустимо, если явно задокументировано).
- Минимально жизнеспособный клиентский контракт: отправить сообщение в канал, получить историю/поток обновлений.
- Персистентность лога и снапшотов на диске для переживания рестартов.

**Non-Goals:**

- Полноценный end-to-end шифрованный мессенджер, федерация, moderation, вложения и поиск.
- Смена алгоритма консенсуса или «Raft по сети поверх произвольного брокера» без явного лога на узлах.
- Полная матрица клиентов (iOS/Android/Desktop) — достаточно контракта API и, при необходимости, одного эталонного клиента позже.
- Политика **TTL / автоматическое удаление** старых сообщений из состояния — не в MVP (только рост вместе с диском и лимитами оператора).

## Decisions

1. **Raft как единственный источник истины для команд**  
   Все изменения домена проходят через лог Raft как сериализованные команды.  
   *Альтернатива:* отдельная БД + best-effort синхронизация — отклонена из-за сложности согласованности.

2. **Hashicorp Raft (`github.com/hashicorp/raft`) vs собственная реализация**  
   Используется **Hashicorp Raft** как встраиваемая библиотека: `FSM`, снапшоты и хранилища лога — через её интерфейсы, с тонким адаптером под домен и транспорт между узлами.  
   *Альтернатива:* учебная реализация с нуля — высокий риск ошибок, не цель MVP.

3. **Транспорт клиента: gRPC (стриминг) + небольшой HTTP health**  
   gRPC даёт явный контракт и стрим для push-обновлений; HTTP `/health` — для оркестраторов.  
   *Альтернатива:* только REST long-poll — проще отладка, хуже модель стриминга; можно добавить позже.

4. **Чтение истории с лидера**  
   В MVP клиент для «свежих» чтений и стриминга ходит на **текущего лидера**; последователи используются для репликации и восстановления.  
   *Альтернатива:* linearizable read через Raft read index — желательно как этап после MVP, если понадобится строгость чтений.

5. **Хранилище**  
   **`LogStore` и `StableStore`:** **BoltDB** через официальный адаптер [`github.com/hashicorp/raft-boltdb`](https://github.com/hashicorp/raft-boltdb) (отдельные файлы БД для лога и стабильного хранилища — по соглашению в коде/конфиге, оба на локальном диске узла).  
   **`SnapshotStore`:** файловый **`raft.FileSnapshotStore`** из [`github.com/hashicorp/raft`](https://github.com/hashicorp/raft) — каталог снапшотов на диске; прикладной снапшот FSM уходит в тот же поток/механизм, что ожидает библиотека.  
   *Альтернатива:* Pebble/Badger/custom файловый лог — если позже упремся в I/O или размер лога на Bolt; *альтернатива:* внешняя SQL только для аналитики — non-goal на старте.

6. **Сериализация команд в записи лога Raft**  
   Полезная нагрузка команд домена (то, что попадает в `Apply` / payload записи лога) кодируется в **Protobuf** — те же `.proto`, что и для gRPC API, единый контракт и кодоген, компактнее JSON. Версионирование схемы команд — через номер/поля в сообщении и дисциплину совместимости при деплое. Для отладки в dev допускается логировать **protojson** рядом с индексом/термом, не меняя формат хранения.  
   *Альтернатива:* JSON в логе — проще ручная отладка, но слабее контракт и больше объём; не выбираем для MVP.

7. **Идемпотентность отправки сообщений**  
   Команда отправки сообщения содержит **опциональное** поле `client_message_id` (непрозрачная строка, например UUID). Если оно **задано**, FSM трактует пару `(channel_id, client_message_id)` как ключ: повторная **закоммиченная** команда с тем же ключом в том же канале **не добавляет** второе видимое сообщение и сходится с тем же исходом, что и первое применение (тот же идентификатор/позиция в истории). Если поле **пустое** — семантика **at-most-once с точки зрения клиента**: при сетевых ретраях без ключа возможны дубликаты в истории; для надёжных клиентов ключ **рекомендуется всегда**. Дубликаты по-прежнему могут порождать **две записи в логе Raft** при повторной отправке с новым ключом — это приемлемый компромисс MVP.  
   *Альтернатива:* обязательный ключ и дедуп только на лидере до `Apply` — сложнее при смене лидера без состояния; *альтернатива:* без идемпотентности вообще — хуже UX при gRPC retries.

8. **Retention сообщений и компактация лога Raft**  
   **Сообщения:** в MVP **нет** политики выселения по возрасту/объёму; история в state machine накапливается (ограничение — диск и эксплуатационные лимиты). Отдельная **компакция «удалить старые сообщения»** — вне MVP.  
   **Лог Raft:** укорочение префикса лога только штатными механизмами **Hashicorp Raft** — **снапшоты** по настраиваемым порогам (**число записей и/или занятый размер** лога), плюс `Snapshot` FSM; «только снапшот без порогов» не используем — пороги задаются конфигом. Файлы снапшотов лежат в каталоге **`FileSnapshotStore`** на диске узла.

## Risks / Trade-offs

- **[Risk] Сложность интеграции FSM и снапшотов** → **Mitigation:** начать с узкого набора команд и одного формата снапшота; property-тесты на детерминизм применения команд.
- **[Risk] Сплит-brain при неправильной конфигурации кворума** → **Mitigation:** жёсткая валидация `cluster` конфигурации, документация по деплою, интеграционные тесты с kill узлов.
- **[Risk] Нагрузка на лидера** → **Mitigation:** батчинг append, лимиты размера сообщения, backpressure в gRPC; позже — read index / follower reads при необходимости.
- **[Trade-off] Упрощённые чтения с лидера** — клиент обязан уметь переключаться при смене лидера (ошибка «not leader» с подсказкой адреса).

## Migration Plan

Первый релиз — **новый сервис** без миграции существующих данных. Порядок выката:

1. Поднять статический кластер (известный список `node_id` + адреса Raft и gRPC).
2. Убедиться в выборе лидера и прохождении health на всех узлах.
3. Включить клиентский трафик только после готовности кворума.

**Rollback:** остановить клиентский ingress, оставить кластер для расследования; откат бинаря на предыдущую версию при сохранении совместимости формата лога/снапшота (версионирование записей лога).
